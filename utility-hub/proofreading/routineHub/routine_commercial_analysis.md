## II. 잘한 부분

### 1. 기능 범위가 넓다 (85점)

일일 계획 → 실행 → 회고 → 주간 리뷰 → 월간 캘린더 → 아카이브까지 **루틴 관리의 전체 생명주기**를 커버하고 있다. 대부분의 경쟁 앱은 이 중 2~3개만 제공한다.

- 계획/확정 2단계 모드 전환은 명확한 의도가 있는 설계
- KPT(Keep-Problem-Try) 기반 회고 프레임워크는 검증된 방법론
- 템플릿 시스템 + 배치 적용은 반복 루틴 사용자에게 실질적 가치 제공
- 캘린더 이벤트의 슬롯 할당 알고리즘은 기술적 완성도가 높음

### 2. Kinetic Timeline (드래그 스케줄링)

타임라인 기반 드래그&드롭 스케줄링은 **차별화 포인트**다. 15분 단위 스냅, 레인 기반 겹침 방지, 리사이즈 등 완성도가 높다. 대부분의 경쟁 앱은 단순 리스트 UI를 사용한다.

### 3. 상태 관리 아키텍처

Zustand 스토어가 잘 구조화되어 있다. 불변성 패턴, 중앙화된 에러 처리, `useMemo`/`useCallback` 최적화가 일관되게 적용됨.

### 4. 백엔드 안정성

- Idempotent한 `getOrCreateTodayPlan()` 설계
- Unique constraint + 중복 복구 로직
- `@Transactional(readOnly = true)` 기본 설정
- EntityGraph 최적화 (아카이브 쿼리)
- 체계적인 ErrorCode enum 기반 에러 처리

### 5. 테마 시스템의 기술적 구현

CSS 변수 기반 테마 전환 + 라이트/다크 모드 분리는 올바른 접근이다. CSS 유틸리티 클래스(`mystic-gradient` 등)로 Tailwind의 CSS 변수 호환 문제를 해결한 것도 실용적.

## III. 부족한 부분 (냉정한 평가)

### 1. 상업적 차별화 부재 (55점) — 가장 치명적

**문제**: "왜 이 앱을 써야 하는가?"에 대한 답이 약하다.

- **경쟁 앱 대비**: Todoist, TickTick, Notion, Structured 등과 비교했을 때 핵심 기능(일일 계획, 타임블록)은 이미 시장에 존재
- **킬러 피처 부재**: Kinetic Timeline이 유일한 차별점인데, 이것만으로는 사용자 전환 동기가 약함
- **타겟 사용자 불명확**: 일반인용? 직장인용? 학생용? 자기계발러용? 페르소나가 정의되지 않았다
- **수익 모델이 빈약**: 테마 판매만으로는 수익화 불가능. 12개 테마 x 100~500P는 인앱결제로 전환해도 매출 한계가 명확

**필요한 것**:
- AI 기반 루틴 추천/최적화 (수면 패턴, 생산성 데이터 기반)
- 소셜 기능 (루틴 공유, 그룹 챌린지)
- 외부 연동 (Google Calendar, Apple Health, 포모도로 타이머)
- 프리미엄 구독 모델 설계

### 2. 사용자 온보딩 부재

**문제**: 첫 사용자가 앱에 들어오면 빈 화면을 마주한다.

- 튜토리얼/가이드 없음
- 샘플 데이터/기본 템플릿 미제공
- Kinetic Timeline의 드래그&드롭을 직관적으로 이해하기 어려움
- "확정" 개념을 설명하는 UI 없음

**영향**: 첫 세션 이탈률이 매우 높을 것으로 예상됨 (60~70%+)

### 3. 모바일 UX가 데스크톱의 축소판 수준

**문제**: 반응형은 되지만 **모바일 네이티브 경험**이 아니다.

- Kinetic Timeline 가로 스크롤은 모바일에서 불편함
- 드래그&드롭이 터치 디바이스에서 정밀도 부족
- TaskAddBar의 폼이 작은 화면에서 복잡
- 슬라이드 패널이 모바일 풀스크린이 아님

**영향**: 루틴 앱의 주 사용 환경은 모바일(아침/저녁)인데, 이 환경에서의 경험이 약하다.

### 4. 데이터 인사이트가 얕다

**문제**: 데이터를 쌓지만 분석이 피상적이다.

- 주간 리뷰: 단순 완료율 차트 + KPT 텍스트 입력
- 월간: 완료율과 XP만 표시
- **없는 것들**: 카테고리별 시간 분석, 습관 형성 추세(streak), 생산적 시간대 분석, 예상 vs 실제 소요 시간 비교, 장기 트렌드 그래프

**경쟁 앱 비교**: Toggl Track, RescueTime 수준의 인사이트가 기대되는 시장이다.

### 5. 퍼포먼스 우려 (65점)

**백엔드**:
- `getShopData()` 호출 시 매번 전체 MonthlyLog → DailyPlan → XP를 재계산 (O(months x plans))
- 월간 상태 조회 시 N+1 쿼리 가능성 (중첩 루프 내 Task 조회)
- 템플릿 목록 페이지네이션 없음
- XP를 캐싱하지 않고 매번 동적 계산

**프론트엔드**:
- 번들 사이즈 867KB (코드 스플리팅 없음)
- 단일 `index.js` 청크 — 초기 로딩 느림
- KineticTimeline의 매 드래그 이벤트마다 전체 리렌더 가능성

### 6. 테스트 부재

프론트엔드/백엔드 모두 **테스트 코드가 보이지 않는다**. 상업 서비스로서는 가장 큰 리스크.

- 단위 테스트 없음
- 통합 테스트 없음
- E2E 테스트 없음
- XP 계산, 권한 검증, 중복 처리 같은 핵심 로직이 테스트 없이 운영되는 것은 위험

### 7. 접근성(Accessibility) 미흡

- 키보드 네비게이션 불완전 (특히 Timeline 드래그 작업)
- 스크린 리더 지원 부족
- 색상만으로 상태를 표현하는 부분 존재 (WCAG AA 미충족)
- 포커스 관리가 모달/패널 전환 시 불명확

### 8. 에러/엣지케이스 처리

- 오프라인 상태 처리 없음
- 낙관적 업데이트(Optimistic Update) 미적용 — 모든 동작이 API 응답 대기
- 동시 수정 충돌 처리 없음 (여러 탭에서 같은 날 수정 시)
- 토큰 만료 시 graceful한 재인증 플로우 불확실

## VI. 기술 스택 요약

### Frontend
- **Framework**: React 18 + TypeScript
- **상태관리**: Zustand
- **스타일링**: Tailwind CSS + CSS Custom Properties (mystic-* 테마 시스템)
- **라우팅**: React Router v6 (RoutineLayout 하위 중첩 라우팅)

### Backend
- **Framework**: Spring Boot + Spring Security (OAuth2)
- **ORM**: Spring Data JPA (Hibernate)
- **DB**: PostgreSQL (추정, JPA + Unique Constraints)
- **인증**: Naver/Google OAuth2
- **API**: REST API 31개 엔드포인트 (Routine 27 + Shop 4)

## VII. 결론

Routine Hub는 **기능적 완성도가 높은 사이드 프로젝트**이다. 일일 계획부터 월간 리뷰까지의 넓은 커버리지와 Kinetic Timeline 같은 인터랙션은 기술적 역량을 잘 보여준다.

하지만 **상업 서비스로서는 아직 부족하다**. 핵심 문제는 기술이 아니라 **"왜 이 앱을 써야 하는가"에 대한 답이 없다는 것**이다. 기능을 더 추가하기보다는:

1. **타겟 사용자를 좁히고** (예: "매일 루틴을 지키고 싶은 20대 직장인")
2. **그 사용자만을 위한 킬러 피처를 만들고** (예: AI 기반 개인 최적화 루틴)
3. **첫 5분 경험을 설계하는 것** (온보딩)

이 세 가지가 지금 가장 시급하다. 기술적 토대는 충분히 탄탄하니, 방향성을 잡으면 빠르게 발전할 수 있는 프로젝트다.
