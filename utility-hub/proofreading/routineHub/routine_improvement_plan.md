## II. 개선 항목별 점수 변화 상세

---

### 1. 타겟 페르소나 확정: "20~30대 직장인"

**영향 카테고리**: 상업적 경쟁력
**점수 변화**: +3점

| 항목 | 초기 상태 | 개선 후 |
|---|---|---|
| 타겟 정의 | 불명확 (일반인? 학생? 직장인?) | 20~30대 직장인 (계획 수립+실행 주 연령대) |
| 마케팅 메시지 | 범용적 → 공감 약함 | "직장인의 하루를 설계하다" 등 타겟 맞춤 가능 |
| 기능 우선순위 | 기준 없음 | 출퇴근 루틴, 업무 카테고리 등 직장인 중심 설계 가능 |

**평가**: 단독 효과는 크지 않으나 이후 모든 의사결정(UX, 기능, 수익 모델)의 **기준점**이 됨. 가장 먼저 확정되어야 할 항목이라 실행 가치가 높다.

---

### 2. 모바일 UX 재설계

**영향 카테고리**: UI/UX (+10), 리텐션 (+5)
**점수 변화**: +12점 (중복 반영 조정)

| 항목 | 초기 상태 | 개선 후 |
|---|---|---|
| 계획 모드 | 데스크톱 Timeline 축소판 | 모바일 전용 리스트 기반 계획 모드 |
| 태스크 완료 | 작은 체크박스 | 원탭 스와이프 완료 |
| 회고 작성 | 데스크톱 폼 그대로 | 모바일 최적화 간소 폼 |
| 전체 경험 | 반응형 (되긴 됨) | 모바일 네이티브 수준 |

**평가**: 루틴 앱의 사용 패턴은 아침(계획 확인) → 낮(실행 체크) → 저녁(회고)이며, 3회 모두 모바일이 메인 디바이스. **DAU 30~40% 증가** 예상. 전체 항목 중 체감 효과가 가장 크다.

---

### 3. 온보딩 + 매뉴얼 (모바일 UX 확정 후)

**영향 카테고리**: 리텐션 (+4), 상업적 경쟁력 (+2)
**점수 변화**: +5점

| 항목 | 초기 상태 | 개선 후 |
|---|---|---|
| 첫 진입 | 빈 화면 | 5단계 이내 가이드 + 기본 템플릿 선택 |
| 기본 템플릿 | 없음 | 직장인 기본 3~4개 (출근 루틴, 업무 집중, 자기계발 등) |
| Kinetic Timeline | 사용법 설명 없음 | 인터랙티브 튜토리얼 or 툴팁 가이드 |
| 첫 세션 이탈률 | 예상 60~70% | 예상 30~40% |

**평가**: 모바일 UX 확정 후 온보딩을 만드는 판단은 정확하다. 바뀔 UI에 맞춰 먼저 만들면 이중 작업이 됨. 단, 기본 템플릿 3~4개는 UX 확정 전에도 넣을 수 있으므로 선행 작업으로 권장.

### 4. Google/Naver 캘린더 단방향 동기화

**영향 카테고리**: 기능 완성도 (+5), 상업적 경쟁력 (+8)
**점수 변화**: +8점 (중복 반영 조정)

| 항목 | 초기 상태 | 개선 후 |
|---|---|---|
| 외부 연동 | 없음 (독립 앱) | OAuth2 기반 단방향 동기화 |
| 사용자 심리 | "기존 캘린더를 버려야 하나?" | "Google Calendar에도 자동으로 뜬다" |
| 시장 신뢰도 | 신규 앱 불안감 | "Google Calendar 연동" = 즉시 신뢰도 상승 |
| 기술 비용 | - | OAuth2 인프라 이미 존재 → 추가 인증 비용 낮음 |

**평가**: **ROI가 가장 높은 기능**. 양방향은 충돌 해결 등 복잡도 10배이므로 단방향이 현실적. 기존 OAuth2 인프라 활용으로 구현 비용 대비 효과가 탁월하다.

**구현 범위 (단방향)**:
- Routine Hub 월간 계획 → Google Calendar 이벤트 생성
- Routine Hub 월간 계획 → Naver Calendar 일정 동기화
- 사용자가 동기화 ON/OFF 선택 가능

---

### 5. AI 사진 인식 → 월간 캘린더 자동 배치

**영향 카테고리**: 상업적 경쟁력 (+5), 기능 완성도 (+3)
**점수 변화**: +5점 (중복 반영 조정)

| 항목 | 초기 상태 | 개선 후 |
|---|---|---|
| 월간 계획 입력 | 수동 입력 only | 사진 촬영 → AI 추출 → 확인/수정 → 배치 적용 |
| 차별화 | Kinetic Timeline만 | + "아날로그 → 디지털 브릿지" 포지셔닝 |
| 바이럴 가능성 | 낮음 | SNS 공유 유도 가능 ("내 손글씨 계획표가 앱으로!") |

**설계 방향 (확정)**:
- 완전 자동 생성 X → "AI 제안 후 사용자 확인" 방식
- 고정 템플릿 + 커스텀 조합 유지 (사람마다 업무 패턴이 다르므로)
- AI는 사진 OCR → 태스크 리스트 추출 → 시간대 제안까지만 담당
- 최종 배치는 사용자가 확인/수정 후 적용

**평가**: 마케팅적으로 강력하나, OCR 정확도가 80% 이하면 오히려 불만 요소. MVP 단계에서는 "인식 → 확인 → 적용" 3단계를 명확히 분리해야 한다.

---

### 6. 알림/리마인더 시스템

**영향 카테고리**: 리텐션 (+5)
**점수 변화**: +4점

| 항목 | 초기 상태 | 개선 후 |
|---|---|---|
| 리마인더 | 없음 (사용자가 앱을 직접 열어야 함) | 단계별 알림 시스템 |
| DAU 의존 | 사용자 의지에 100% 의존 | 알림이 앱 재방문 유도 |

**웹+앱 하이브리드 환경 고려 알림 전략**:

| 단계 | 방식 | 웹 | 앱 | 사용자 체감 | 구현 난이도 |
|---|---|---|---|---|---|
| 1단계 | 이메일 데일리 다이제스트 | O | O | 중간 | 낮음 |
| 2단계 | Web Push (Service Worker) | O | O | 높음 | 중간 |
| 3단계 | 네이티브 Push (FCM) | X | O | 최고 | 높음 |
| 고려 | 카카오톡 알림톡 | O | O | 높음 | 중간 (API 비용) |

**평가**: 한국 20~30대 직장인 타겟이라면 카카오톡 알림톡도 도달률 측면에서 고려 가치 있음. 웹+앱 하이브리드 특성상 방식 선택이 중요하므로 신중한 판단이 필요한 영역.

### 7. 테스트 코드 작성 (AI 에이전트 활용)

**영향 카테고리**: 코드 품질 (+7)
**점수 변화**: +7점

| 항목 | 초기 상태 | 개선 후 |
|---|---|---|
| 단위 테스트 | 없음 | AI 에이전트로 핵심 로직 커버 |
| 통합 테스트 | 없음 | API 엔드포인트 통합 테스트 |
| 테스트 대상 | - | XP 계산, 권한 검증, 중복 처리, 포인트 차감 |
| 배포 자신감 | 낮음 (수동 검증 의존) | 높음 (자동화된 회귀 테스트) |

**AI 에이전트 활용이 적합한 이유**:
- 테스트 코드는 규칙이 명확하고 반복적 → AI가 가장 효율적으로 처리 가능
- 엔티티/서비스/컨트롤러 패턴이 일관적 → 템플릿화 용이
- 사람이 직접 작성하는 것보다 빠르고, 커버리지 누락이 적음

**우선 테스트 대상**:
1. `XpCalculator` - 포인트 계산 정확성
2. `ShopService` - 구매/권한/어드민 바이패스 로직
3. `RoutineService` - 플랜 생성/확정/중복 처리
4. `RoutineController` - API 응답 형식 및 인증 검증

---

### 8. 퍼포먼스 최적화 (AI 에이전트 활용)

**영향 카테고리**: 퍼포먼스/확장성 (+12)
**점수 변화**: +10점 (중복 반영 조정)

| 항목 | 초기 상태 | 개선 후 |
|---|---|---|
| 번들 사이즈 | 867KB 단일 청크 | React.lazy 코드 스플리팅 |
| XP 계산 | 매 요청마다 전체 재계산 | 캐싱 (MonthlyLog.totalXp 활용) |
| N+1 쿼리 | 월간 조회 시 중첩 루프 | EntityGraph / Fetch Join 최적화 |
| 초기 로딩 | 느림 (전체 번들 다운로드) | 라우트별 분리 → 50% 이상 감소 |

**AI 에이전트 활용이 적합한 이유**:
- 코드 스플리팅 → 패턴이 정형화됨 (`React.lazy` + `Suspense`)
- N+1 해결 → JPA 쿼리 분석 + `@EntityGraph` 적용은 기계적 작업
- XP 캐싱 → 기존 `MonthlyLog.totalXp` 필드 활용하여 write-through 캐시 구현

## III. 수익 모델 (검토 중)

현재 수익 모델은 **테마 판매(포인트 기반)** 단일 구조. 개선 계획에는 포함되지 않았으나, 90점 이상을 넘기기 위해 향후 검토가 필요한 영역.

**현재 인프라에서 자연스럽게 확장 가능한 방향**:

| 모델 | 기존 인프라 활용도 | 직장인 타겟 궁합 | 수익 잠재력 |
|---|---|---|---|
| 프리미엄 테마 실결제 | Shop 시스템 그대로 | 보통 | 낮음 (단독 한계) |
| 월 구독 Pro | AI 기능 + 캘린더 동기화 + 고급 인사이트를 Pro 전용으로 | 높음 | **높음 (가장 현실적)** |
| PDF 리포트 내보내기 | 주간/월간 데이터 이미 존재 | 높음 (직장인 보고용) | 중간 |
| 팀/기업 플랜 | 사용자 시스템 확장 필요 | 높음 | 높음 (장기) |

**결정 보류 사유**: 핵심 기능(모바일 UX, 캘린더 동기화, AI) 완성 후 사용자 데이터 기반으로 판단하는 것이 합리적.

## VI. 실행 로드맵

```
Phase 1 (즉시)
├── 타겟 페르소나 문서화
├── 기본 템플릿 3~4개 추가 (출근 루틴, 업무 집중, 자기계발, 휴식일)
└── 테스트 코드 작성 시작 (AI 에이전트) ← 병행

Phase 2 (1~2개월)
├── 모바일 UX 재설계 + 구현
├── 퍼포먼스 최적화 (AI 에이전트) ← 병행
│   ├── 코드 스플리팅 (React.lazy)
│   ├── XP 캐싱
│   └── N+1 쿼리 해결
└── 알림 시스템 방식 검토 및 결정

Phase 3 (2~3개월)
├── Google/Naver 캘린더 단방향 동기화
└── 온보딩 플로우 구현 (모바일 UX 확정 후)

Phase 4 (3~5개월)
├── AI 사진 인식 → 월간 배치 (MVP)
├── 알림 시스템 구현 (1단계: 이메일 → 2단계: Web Push)
└── 수익 모델 검토 및 결정

Phase 5 (5~6개월+)
├── 수익 모델 구현 (구독 Pro 등)
├── 데이터 인사이트 고도화 (카테고리별 분석, streak, 트렌드)
└── 접근성(A11y) 개선
```
